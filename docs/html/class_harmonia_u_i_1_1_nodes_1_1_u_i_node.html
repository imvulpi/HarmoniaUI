Detailed Description

Base class for all HarmoniaUI nodes, extending Godot's Control.

UINode serves as the foundation for all HarmoniaUI UI elements and can be applied to any Godot Control node.
It provides a unified styling, layout, and rendering system on top of Godot's native UI, making it possible to use HarmoniaUI features with both Harmonia and native Godot controls.

A UINode can be used directly or subclassed The Parent property ensures parent references always point to a UINode instance or null.

Key features:

  • CSS styling system with StyleResource. (But with Types)
  • Parsed and computed styles.
  • Separate content size (ContentWidth, ContentHeight) from actual node size.
  • Customizable layout via ILayoutEngine.
  • Customizable rendering via IVisualEngine.
  • Designed for extensibility (custom input and animations are planned).

The StyleResource is also exported to the Godot editor, giving designers easy, direct control over visuals and layout without code changes.

UINode supports an almost instant live preview in the editor, updating at 50 ms intervals by default (intervalMs = 50).
This preview is only active in Debug mode and inside the editor, and is disabled in Release builds for performance and reduced overhead.

Creating Custom Nodes

Extending UINode is straightforward, but you should ensure that overridden methods either call their base implementation or provide equivalent functionality, so that the styling, layout, and rendering systems remain functional.

The following members can be overridden:

  • UpdateLayout
  • _Draw
  • _Ready
  • _EnterTree
  • _ExitTree
  • ContentWidth
  • ContentHeight

Remember: ContentWidth and ContentHeight are virtual, They're used to set Size for children nodes, 100% in children style is 100% of ContentWidth or ContentHeight

To extend the behavior of UINode, you can create custom LayoutResource or VisualResource classes and implement corresponding ILayoutEngine or IVisualEngine instances.
This allows you to define entirely new layouts or rendering that integrate seamlessly with the HarmoniaUI styling and node system.

When to Add New Nodes vs. Extend Layout/Visual Engines

In HarmoniaUI, deciding whether to create a new UINode subclass or extend layout/visual engines depends on your customization:

  • Add a new node subclass when you need new interactive behaviors, unique properties, or specific control logic tied to a UI element. For example, buttons, sliders, or custom containers.
  • Extend layout engines to implement new ways of measuring, positioning, or sizing nodes. This is ideal when you want to change layout behavior but not the individual node's logic. it can apply to all UINode
  • Extend visual engines when you want to change how nodes are rendered, such as custom drawings, effects, or new styles that apply across multiple node types, beause it can apply to all UINode

◆ UpdateLayout()

virtual void HarmoniaUI.Nodes.UINode.UpdateLayout ( )
inlinevirtual

Recomputes the node's style and layout, applies size and position changes, and triggers a redraw of the node and its children.

This method computes the current ParsedStyle, invokes the configured ILayoutEngine to compute sizes and arrange the node and its children, to finally call for a redraw

Updates layouts of all direct children which essentially means it updates all children below this UINode in the hierarchy

Since layout changes affect rendering, UpdateLayout also calls for a redraw via Godot's QueueRedraw(); call.

Typically called after style changes, adding/removing children, viewport change or when a manual layout refresh is needed.
